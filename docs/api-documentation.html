§ 3 · Authentication

Authentication

Both APIs use static API key authentication via HTTP headers. No OAuth or session tokens required.

OLT Provisioning API
HTTP Header
X-API-Key: <your-api-key>

Configured via OLT_API_KEYS environment variable (comma-separated for multiple keys). Missing or invalid key returns 401 Unauthorized.

M6K Circuit API
HTTP Header
X-Api-Key: JTL-BSS-2026-M6KPROV
Security note: Both APIs are deployed on an internal network segment (172.16.37.18). Ensure the server is not exposed to the public internet without a TLS-terminating reverse proxy (nginx).
Common Response Envelope
JSON
{
  "success": true,
  "data": { /* endpoint-specific payload */ }
}

§ 4.1 · OLT API

Provision ONU

Single endpoint executes the complete 14-step GPON provisioning workflow.

POST /provision Full ONU provisioning — authorize → T-CONT → GEM → service-port → ACS → WiFi → notify
Provisions a customer's ONU end-to-end. The API locates the ONU by serial number on the OLT, assigns the next available ONU ID, runs all GPON configuration commands, registers the ONU with ACS (TR-069), waits for the ONU to phone home (up to 120s), pushes WiFi credentials via ACS, saves to database, and sends SMS/Email/WhatsApp notifications.
Request Body
FieldTypeRequiredDescription
customer_idstringrequiredUnique customer identifier, e.g. CUST-0012345
customer_namestringrequiredFull name for notification messages
customer_phonestringoptionalPhone for SMS/WhatsApp delivery
customer_emailstringoptionalEmail for credential delivery
onu_serial_numberstring(16)requiredONU serial, e.g. ZTEG12345678
onu_modelstringoptionalONU type profile on OLT. Default: F660
olt_idstringrequiredOLT name from inventory, e.g. OLT-NBI-01
package_idstringrequiredBandwidth package, e.g. GPON-10M
service_vlanintrequiredCustomer VLAN tag (C-VLAN / inner tag)
oam_vlanintoptionalOAM management VLAN. Default: 1450
svlanintoptionalService VLAN (S-VLAN) for Q-in-Q. Omit for single-tagged.
descriptionstringoptionalONU description tag on OLT

§ 4.2 · OLT API

Lifecycle Operations

Suspend, activate, change package, and deprovision endpoints — all identified by customer_id.

DELETE /provision/{customer_id} Remove ONU from OLT and database
Runs no onu {id} on the OLT port and removes all service-port and T-CONT config. Deletes ONU record from database. Irreversible.
curl
curl -X DELETE http://172.16.37.18:8000/provision/CUST-0012345 \
  -H "X-API-Key: your-key"

Returns 204 No Content on success.

PUT /provision/{customer_id}/suspend Suspend — block all ONU traffic
Sets ONU admin-state to disable on the OLT. Config is preserved; reactivation restores service instantly.
Response
{ "success": true, "data": { "customer_id": "CUST-0012345", "state": "suspended" } }
PUT /provision/{customer_id}/activate Activate — restore ONU traffic
Sets ONU admin-state to enable. Used after suspension (payment received) or initial test provisioning.
PUT /provision/{customer_id}/package Change bandwidth package — live T-CONT reconfiguration
Reconfigures the T-CONT DBA profile on the OLT to match the new package bandwidth. Takes effect immediately without disrupting existing sessions.
Request Body
{ "package_id": "GPON-50M" }
Response
{ "success": true, "data": { "customer_id": "CUST-0012345", "package_id": "GPON-50M", "bandwidth_kbps": 51200 } }
§ 4.3 · OLT API

ONU Discovery

Query the OLT in real time for unconfigured ONUs waiting to be provisioned.

GET /onu/{olt_id}/unconfigured List unregistered ONUs on a specific PON port
Runs show gpon onu uncfg gpon-olt_{f}/{s}/{p} on the OLT and returns parsed ONU serial numbers and status. Use to confirm ONU is visible before provisioning.
ParameterTypeDefaultDescription
olt_idstring (path)—OLT name, e.g. OLT-NBI-01
frameint (query)1OLT chassis frame
slotint (query)7Line card slot
portint (query)2GPON port number
curl
curl "http://172.16.37.18:8000/onu/OLT-NBI-01/unconfigured?slot=7&port=2" \
  -H "X-API-Key: your-key"
GET /onu/{olt_id}/find/{sn} Find ONU by serial number — scans all known PON ports
Scans all GPON ports on slots 7 and 9 (ports 0–15) to locate an ONU by its serial number. Returns the port location when found. Used by the provisioning flow internally, and by BSS to verify ONU connectivity.
curl
curl "http://172.16.37.18:8000/onu/OLT-NBI-01/find/ZTEGABC12345" \
  -H "X-API-Key: your-key"

§ 4.5 · OLT API

14-Step Provisioning Flow

Sequence of operations executed by POST /provision — from BSS request to customer online with WiFi.

Step-by-Step Breakdown
1
Resolve OLT by name
Look up OLT record in DB by the string name (e.g. OLT-NBI-01). 404 if not found.
2
Duplicate check
Verify customer_id has no existing ONU in DB. Returns 409 Conflict if already provisioned.
3
Locate ONU by serial
Scans GPON ports (slots 7+9, ports 0–15) with show gpon onu uncfg. 422 if not found.
show gpon onu uncfg gpon-olt_1/7/2
4
Determine next ONU ID
Parses show running-config for the port to find highest existing ONU ID + 1.
5
Authorize ONU
Binds ONU serial to an ONU ID slot on the port.
onu 5 type F660 sn ZTEGABC12345
6
Configure T-CONT + GEM port
Sets DBA profile (bandwidth) via T-CONT and links to GEM port.
tcont 1 profile Fix_10M  ·  gemport 1 tcont 1
7
Create service port
Maps Ethernet UNI to service VLAN. Supports Q-in-Q (svlan) for double-tagging.
service-port 1 ethuni eth_0/1 user-vlan 200 svlan 200
8
Configure ACS endpoint
Pushes TR-069 ACS URL, username, and password to ONU over CLI.
tr069-serv-url http://197.232.61.253:7547
9
Save OLT config
Issues write command to persist configuration.
10
Save ONU to database
Persists ONU record, WiFi credentials, customer info, and location to MariaDB.
11
Wait for TR-069 Inform
Polls GenicACS every 5s for up to 120s until ONU phones home. Non-blocking if timeout.
GET /devices?query={"_id":"ZTEG..."}&projection=_lastInform
12
Push WiFi via ACS (TR-181)
Sets SSID and passphrase on both 2.4GHz and 5GHz bands via GenicACS task API.
Device.WiFi.SSID.1.SSID · Device.WiFi.AccessPoint.1.Security.KeyPassphrase
13
Send customer notifications
Delivers WiFi SSID + password via SMS (Africa's Talking), Email (SMTP), and WhatsApp.
14
Return ProvisionResponse
Returns 201 with ONU location, WiFi SSIDs, ACS status, notifications sent, and duration.

§ 5.1 · M6K API

Create VPWS Circuit

VPWS pseudowire between two PE routers. Auto-assigns VC ID, pushes xconnect to both ends, polls PW state.

POST /circuit/vpws Create VPWS pseudowire — xconnect on A+Z PEs, poll PW UP
Validates the circuit ID is unique, auto-assigns a VC ID from pool 5000–9999, SSHs to both A-end and Z-end PE routers and pushes xconnect configuration, then polls PW state for up to 90s. Supports both untagged and VLAN-tagged (sub-interface) attachment circuits.
Request Body

a_end.interfacestringrequiredPhysical interface, e.g. GigabitEthernet3/0/1

a_end.vlanintoptionalVLAN tag for dot1q encapsulation

§ 5.3 · M6K API

Manage Circuits

Delete, suspend, activate, and query live PW state for existing circuits.

DELETE /circuit/{circuit_id} Delete — remove xconnect from both PEs
Issues no xconnect on both A-end and Z-end PEs. For VLAN-tagged circuits, also removes the sub-interface. Returns 204.
ZXR10 CLI (A-end)
interface GigabitEthernet3/0/1.100
 no xconnect 10.0.0.3 5001 pw-class default
!
no interface GigabitEthernet3/0/1.100
PUT /circuit/{circuit_id}/suspend Suspend — shutdown A-end AC interface
Issues shutdown on the A-end attachment circuit interface. The xconnect config is preserved; reactivation is instantaneous.
ZXR10 CLI
interface GigabitEthernet3/0/1
 shutdown
GET /circuit/{circuit_id}/pw-state Live PW state from router
SSHs to the A-end PE and runs show mpls l2vpn vc {vc_id}. Parses output for UP or DOWN state. VPWS circuits only.
JSON Response
{
  "circuit_id": "P2P-00001234",
  "vc_id":      5001,
  "pw_state":  "UP",
  "router":    "M6K-NBI-PE1"
}
GET /circuits List all circuits — optional customer_id filter
Returns all circuit records from the in-memory store. Use ?customer_id=CORP-00234 to filter. Used by BSS for reconciliation.
§ 5.4 · M6K API

VPWS Circuit Flow

Sequence of operations for POST /circuit/vpws — from BSS request to live PW.

Error: Rollback on PE Failure
Partial failure handling: If the A-end SSH commands succeed but Z-end fails, the service rolls back by issuing no xconnect on the A-end before returning a 500 error. Circuit IDs are not consumed on failure.
